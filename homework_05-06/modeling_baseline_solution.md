# Домашнее задание №5–6. Baseline-решение, ML-метрики и улучшенные модели

## Проект

**Hotel Booking Cancellation Risk** — ML-сервис для revenue-менеджера отеля. Сервис заранее оценивает вероятность отмены бронирования, ранжирует бронирования по уровню риска и помогает менеджеру работать не со всей базой заявок, а с приоритетным списком бронирований, по которым есть смысл принять управленческое действие.

Главная продуктовая цепочка проекта:

```text
данные бронирования
→ прогноз вероятности отмены
→ оценка ожидаемой финансовой потери
→ бизнес-приоритизация
→ рекомендованное действие менеджера
```

В рамках ДЗ №5–6 построен baseline-пайплайн, проведено сравнение простых и усложненных моделей, выбраны ML- и ranking-метрики, а также добавлен продуктовый слой, который переводит вероятность отмены в бизнес-приоритет.

---

## 1. Постановка ML-задачи

Задача формулируется как бинарная классификация:

```text
y = is_canceled

0 — бронирование реализовалось
1 — бронирование отменено / не реализовалось
```

Модель возвращает вероятность отмены:

```text
risk_score = P(is_canceled = 1)
```

В продукте эта вероятность не используется как автоматическое решение “отменит / не отменит”. Revenue-менеджеру нужен ранжированный список бронирований, требующих внимания. Поэтому задача рассматривается как ranking-first: модель должна поднять потенциальные отмены в верхние 10–20% списка.

---

## 2. Данные и подготовка

В работе используется датасет гостиничных бронирований **Hotel Booking Demand Dataset**. Каждая строка соответствует одному бронированию, целевая переменная — `is_canceled`.

В ноутбуке реализована следующая подготовка данных:

1. Загрузка исходного файла `data/raw/hotel_bookings.csv`.
2. Очистка пропусков и невалидных значений.
3. Удаление невалидных бронирований без гостей.
4. Обработка отрицательного `adr`.
5. Создание даты прибытия `arrival_date`.
6. Сортировка данных по времени.
7. Time-based split на train / validation / test.

Использование временного разбиения важно, потому что в реальном продукте модель обучается на прошлых бронированиях и применяется к будущим. Такой split лучше имитирует production-сценарий, чем случайное разбиение.

## 3. Leakage audit

Для корректной оценки исключены признаки, которые могут быть известны только после отмены, после заезда или после операционных изменений бронирования.

Исключенные признаки:

| Признак                   | Причина исключения                                  |
| ------------------------- | --------------------------------------------------- |
| `reservation_status`      | Напрямую содержит финальный статус бронирования     |
| `reservation_status_date` | Содержит дату финального статуса / отмены           |
| `assigned_room_type`      | Может быть известен после операционных изменений    |
| `booking_changes`         | Может изменяться после создания бронирования        |
| `agent`                   | Высококардинальный анонимизированный operational id |
| `company`                 | Много пропусков и высокий риск нестабильности       |

Также не используются как признаки служебные поля, идентификаторы, split-метки и raw-поля, которые не должны участвовать в прогнозе.

## 5. Продуктовая логика: риск отмены ≠ финансовый риск

Важная бизнес-доработка состоит в том, что `risk_score` не приравнивается к финансовому риску.

```text
risk_score = вероятность отмены
business_priority_score = управленческий приоритет с учетом риска, стоимости брони и типа депозита
```

Формула бизнес-слоя:

```text
booking_value = adr × total_nights

expected_loss = risk_score × booking_value × deposit_loss_factor

business_priority_score = expected_loss
```

Использованные коэффициенты:

| deposit_type | deposit_loss_factor | Бизнес-логика                                                                          |
| ------------ | ------------------: | -------------------------------------------------------------------------------------- |
| `No Deposit` |                1.00 | Отель слабо защищен, отмена может привести к полной потере потенциальной выручки       |
| `Refundable` |                0.80 | Риск частично сохраняется, особенно если отмена поздняя                                |
| `Non Refund` |                0.05 | Вероятность отмены может быть высокой, но финансовая потеря для отеля существенно ниже |

Итог: top-list для менеджера сортируется по `business_priority_score`, а не по сырому `risk_score`.

## 7. Итоговый вывод

Финальная модель — **LightGBM random search**. Она показала лучший `quality_index = 0.7607` и выполнила ключевые целевые ориентиры:

```text
ROC-AUC = 0.8530
PR-AUC = 0.7942
Precision@20 = 0.8172
Recall@20 = 0.4147
Lift@20 = 2.0730
```

Главный продуктовый результат — `Lift@20 = 2.0730`. Это означает, что верхние 20% бронирований, выбранные моделью, содержат отмены примерно в 2.07 раза чаще, чем случайная выборка такого же размера.

`Precision@20 = 0.8172` означает, что среди верхних 20% броней по риску около 82% действительно относятся к отменам. Для revenue-менеджера это важнее, чем accuracy: менеджер работает с ограниченным top-list, а не со всеми бронированиями.

Модель также существенно превосходит rule-based baseline:

```text
LightGBM Lift@20 = 2.0730
Rule-based Lift@20 = 1.3347
```

Это показывает, что ML-модель дает дополнительный сигнал поверх ручной бизнес-логики.

Главный вывод проекта:

```text
Для продукта недостаточно предсказывать вероятность отмены.
Нужно переводить вероятность в управленческий приоритет:
risk_score → expected_loss → business_priority_score → recommended_action.
```
