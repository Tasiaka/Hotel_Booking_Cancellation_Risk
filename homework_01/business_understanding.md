# Домашнее задание №1. Презентация задачи и выбор датасета

## Проект: Hotel Booking Cancellation Risk

**Тема проекта:** ML-сервис для прогнозирования отмен гостиничных бронирований.

**Выбранный датасет:** Hotel Booking Demand Dataset - https://www.kaggle.com/datasets/jessemostipak/hotel-booking-demand

**Тип задачи:** бинарная классификация.

**Целевая переменная:** `is_canceled`.

**Планируемый продукт:** интерактивный сервис/дашборд для revenue-менеджера отеля, который оценивает риск отмены каждого бронирования, ранжирует бронирования по уровню риска и помогает заранее принять управленческие меры.

---

## 0. Краткое описание проекта

Отмена бронирований — значимая проблема для гостиничного бизнеса. Если клиент отменяет бронирование, отель может потерять выручку, получить незаполненный номер и некорректно спланировать загрузку, персонал, закупки и ценовую политику. Особенно критична ситуация, когда отмена происходит поздно и номер уже сложно перепродать.

Цель проекта — разработать прототип ML-сервиса, который заранее оценивает вероятность отмены бронирования. Сервис должен помогать revenue-менеджеру и менеджеру бронирований выявлять заявки с повышенным риском отмены и принимать действия до наступления негативного события.

Продуктовая идея заключается не только в обучении модели, но и в создании прикладного инструмента: таблицы/дашборда с риск-скорингом бронирований, объяснением ключевых факторов риска и расчетом возможного бизнес-эффекта.

## 1. Бизнес-анализ (Business Understanding)

### 1.1. Цель и скоуп проекта

#### Бизнес-проблема

Гостиница получает бронирования через разные каналы: сайт, онлайн-агентства, туристические операторы и корпоративные каналы. Часть клиентов в дальнейшем отменяет бронирование. Это создает несколько бизнес-проблем:

- снижается фактическая загрузка отеля;
- отель может недополучить выручку;
- ухудшается точность операционного планирования;
- сложнее планировать персонал, уборку, питание и дополнительные услуги;
- решения об overbooking принимаются интуитивно, без количественной оценки риска;
- менеджеры тратят одинаковое внимание на все бронирования, хотя риск отмены у них разный.

#### Бизнес-цель проекта

Основная бизнес-цель проекта — **снизить финансовые и операционные потери от отмен бронирований за счет раннего выявления бронирований с высоким риском отмены**.

В рамках учебного проекта бизнес-цель формулируется как создание прототипа сервиса, который:

1. рассчитывает вероятность отмены для каждого бронирования;
2. ранжирует бронирования по уровню риска;
3. выделяет бронирования, требующие внимания менеджера;
4. помогает оценить потенциальные потери выручки;
5. предлагает базовые управленческие действия в зависимости от категории риска.

#### Скоуп первой версии продукта

В первую версию продукта входит:

- загрузка датасета с историей бронирований;
- обучение модели бинарной классификации;
- расчет вероятности отмены бронирования;
- разделение бронирований на категории риска;
- простая интерпретация факторов, влияющих на прогноз;
- расчет технических метрик качества модели;
- прототип дашборда или интерфейса для демонстрации результата.

В первую версию продукта не входит:

- интеграция с реальной PMS/CRM-системой отеля;
- автоматическое изменение цен;
- автоматическая отмена или подтверждение бронирований;
- полноценный A/B-тест на реальных клиентах;
- юридическая проверка бизнес-действий, связанных с депозитами или невозвратными тарифами.

### 1.3. Существующие решения и их ограничения

В реальной гостиничной практике отмены бронирований часто оцениваются на основе простых правил или экспертного опыта. Например, менеджер может считать более рискованными бронирования:

- с большим `lead_time`;
- без депозита;
- от новых клиентов;
- из отдельных каналов продаж;
- с историей предыдущих отмен;
- с низким количеством специальных запросов;
- с гибкими условиями отмены.

Такие правила понятны, но имеют ограничения:

1. **Правила не учитывают сложные комбинации факторов.** Например, сам по себе длинный срок до заезда может быть не критичным, но в сочетании с каналом продаж, типом депозита и историей клиента риск может существенно возрастать.

2. **Экспертная оценка плохо масштабируется.** Менеджер не может вручную анализировать тысячи бронирований.

3. **Нет единого численного риска.** Без ML-модели сложно ранжировать бронирования и выбирать top-N наиболее рискованных заявок.

4. **Нет оценки качества решений.** Простые правила трудно валидировать на исторических данных.

5. **Решение не связано с бизнес-метрикой.** Даже если менеджер понимает, что часть заявок рискованная, не всегда ясно, какую потенциальную выручку можно защитить.

Текущее учебное решение должно устранить эти ограничения за счет модели машинного обучения и продуктового слоя.

## 2. Выбор датасета

### 2.1. Описание датасета

Для проекта выбран датасет **Hotel Booking Demand Dataset**.

Источник данных:

- научная публикация: *Hotel booking demand datasets*, Data in Brief;
- публичная версия датасета доступна на Kaggle.

Датасет содержит данные о бронированиях двух типов отелей:

- `Resort Hotel`;
- `City Hotel`.

Каждая строка соответствует одному бронированию. В данных есть как завершенные, так и отмененные бронирования. В исходном описании указано, что датасеты содержат 40 060 наблюдений для resort hotel и 79 330 наблюдений для city hotel, то есть суммарно 119 390 бронирований. Данные относятся к бронированиям с датами прибытия с 1 июля 2015 года по 31 августа 2017 года.

### 2.2. Почему датасет подходит для проекта

Датасет подходит для MFDP по нескольким причинам:

1. **Есть понятная бизнес-задача.** Отмена бронирования напрямую связана с выручкой, загрузкой и операционным планированием.

2. **Есть целевая переменная.** Поле `is_canceled` позволяет сформулировать задачу бинарной классификации.

3. **Есть достаточно признаков для анализа.** В данных есть информация о сроке до заезда, длительности проживания, типе отеля, канале продаж, типе клиента, депозите, истории отмен и специальных запросах.

4. **Можно сделать сервис, а не только модель.** Результат модели можно использовать в дашборде для менеджера.

5. **Можно оценивать не только ML-качество, но и бизнес-эффект.** Например, считать потенциальную выручку бронирований с высоким риском отмены.

6. **Данные обезличены.** Это снижает риски, связанные с персональными данными.

### 2.4. Признаки, требующие проверки на утечку данных

В проекте важно определить момент времени, в который модель делает прогноз. Предполагаемый момент прогноза — после создания бронирования, но до наступления даты заезда и до факта отмены.

Поэтому нужно проверить и, вероятно, исключить признаки, которые могут быть известны только после изменения статуса бронирования:

| Поле | Возможная проблема |
|---|---|
| `reservation_status` | Напрямую отражает итоговый статус бронирования |
| `reservation_status_date` | Может содержать информацию о факте отмены |
| `assigned_room_type` | Может быть известен не всегда на момент раннего прогноза |
| `booking_changes` | Может изменяться после создания бронирования |
| `days_in_waiting_list` | Нужно проверить доступность признака на момент прогноза |

Основной риск здесь — data leakage. Если оставить признаки, которые становятся известны только после отмены, модель покажет высокое качество на тесте, но будет бесполезна в реальном применении.

## 5. Решаемые задачи с точки зрения аналитики (Data Mining Goals)

### 5.1. Постановка ML-задачи

Задача формулируется как бинарная классификация:

```text
y = is_canceled
```

Где:

```text
0 — бронирование не отменено
1 — бронирование отменено
```

Модель должна возвращать вероятность отмены:

```text
P(is_canceled = 1)
```

Вероятность отмены будет использоваться для ранжирования бронирований по риску.

### 5.2. Бизнес-интерпретация результата модели

Вместо жесткого ответа "отменит / не отменит" сервис должен показывать категорию риска:

| Вероятность отмены | Категория риска | Возможное действие |
|---:|---|---|
| 0.00–0.30 | Низкий риск | Без дополнительных действий |
| 0.30–0.60 | Средний риск | Отправить напоминание или запрос подтверждения |
| 0.60–0.80 | Высокий риск | Проверить условия бронирования, предложить депозит или невозвратный тариф |
| 0.80–1.00 | Критический риск | Учесть в overbooking-стратегии, передать на ручную проверку |

Точные пороги будут уточняться после анализа распределения вероятностей и оценки ошибок модели.

### 5.3. Метрики качества модели

Для оценки результата моделирования будут использоваться следующие метрики:

| Метрика | Зачем используется |
|---|---|
| ROC-AUC | Общая способность модели разделять отмененные и неотмененные бронирования |
| PR-AUC | Качество модели на положительном классе, особенно если есть дисбаланс |
| Recall по классу `1` | Доля фактических отмен, найденных моделью |
| Precision по классу `1` | Доля действительно отмененных среди бронирований, отмеченных как рискованные |
| F1-score | Баланс между Precision и Recall |
| Precision@K | Качество top-K бронирований с наибольшим риском |
| Lift@K | Насколько лучше модель выбирает рискованные бронирования по сравнению со случайным отбором |
| Business value simulation | Оценка потенциальной защищенной выручки при обработке top-N рискованных бронирований |

Accuracy не выбирается основной метрикой, потому что она может быть недостаточно информативной при несбалансированных классах и не отражает ценность выявления именно отмен.

### 5.4. Критерии успешности

Минимальные критерии успешности модели:

| Уровень | Критерий |
|---|---|
| Минимально приемлемый результат | ROC-AUC >= 0.75 |
| Хороший результат | ROC-AUC >= 0.80 |
| Минимально полезный Recall | Recall по классу отмен >= 0.70 при контролируемом Precision |
| Продуктовый критерий | Top-20% бронирований по риску содержит заметно больше отмен, чем случайная выборка |
| Бизнес-критерий | Симуляция показывает потенциальное снижение потерь или улучшение планирования загрузки |

### 5.5. Бизнес-метрика

Для учебного проекта предлагается использовать приближенную бизнес-метрику:

```text
Expected Protected Revenue = Sum(ADR * total_nights) for high-risk bookings
```

Где:

```text
total_nights = stays_in_weekend_nights + stays_in_week_nights
```

Эта метрика показывает объем выручки, связанной с бронированиями высокого риска. Она не означает, что вся эта сумма будет сохранена, но позволяет оценить масштаб бронирований, требующих внимания.

Более точная версия бизнес-метрики может учитывать:

- вероятность отмены;
- стоимость ночи;
- длительность проживания;
- вероятность перепродажи номера;
- стоимость действия менеджера;
- эффект от напоминания или депозита;
- штраф за ошибочное вмешательство в бронирование с низким реальным риском.

## 8. Итоговая формулировка проекта для README

Проект **Hotel Booking Cancellation Risk** посвящен разработке ML-сервиса для прогнозирования отмен гостиничных бронирований. Сервис предназначен для revenue-менеджера отеля и помогает заранее выявлять бронирования с высоким риском отмены, ранжировать заявки по уровню риска и оценивать потенциальные потери выручки.

ML-задача формулируется как бинарная классификация: модель прогнозирует вероятность отмены бронирования на основе параметров бронирования, клиента, канала продаж, истории предыдущих отмен, типа депозита и характеристик поездки.

Планируемый результат проекта — интерактивный дашборд, в котором пользователь может загрузить данные по бронированиям, получить риск-скоринг, увидеть ключевые факторы риска и оценить бизнес-эффект от обработки бронирований с высоким риском отмены.
