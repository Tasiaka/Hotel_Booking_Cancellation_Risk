# UI roles

В MVP реализовано простое разделение интерфейса по ролям для демонстрации ДЗ №8.

## Логины и регистрация

```text
user / user     — рабочий интерфейс менеджера
admin / admin   — техническая админская часть
```

На странице входа есть два режима: **Вход** и **Регистрация**. Новый зарегистрированный пользователь получает роль менеджера. Это сделано для демонстрации пользовательского сценария без выдачи админских прав через UI.

Локальное хранилище зарегистрированных пользователей:

```text
storage/users.json
```

Пароли зарегистрированных пользователей сохраняются как PBKDF2-SHA256 hash с солью. Демо-пользователи `admin/admin` и `user/user` остаются встроенными для проверки ролей.

## Менеджер

Менеджерский интерфейс сделан как рабочий инструмент:

1. выбрать demo dataset или загрузить CSV;
2. нажать «Запустить анализ»;
3. посмотреть 4 главных показателя;
4. открыть список броней к проверке;
5. открыть карточку конкретной брони;
6. посмотреть историю сохраненных batch-запусков.

Технические JSON-ответы, endpoint-ы, логи и детали схемы скрыты от менеджера.

## Админ

Админ видит технический контур:

- статус API;
- активную модель;
- тип СУБД;
- batch-историю;
- схему входных данных;
- список REST endpoint-ов;
- локальные команды запуска;
- UI-логи;
- список demo- и зарегистрированных пользователей.

Это разделение сделано для демонстрации MVP: пользовательский экран не перегружен, но все технические требования задания остаются проверяемыми.

## Логика рабочего списка менеджера

Менеджерский список “К проверке” отсортирован по ожидаемой денежной потере, а не по одной вероятности отмены. Брони `Non Refund` получают прогноз модели как и все остальные, но не попадают в основной рабочий список менеджера, если ожидаемые потери равны нулю. Полный технический просмотр данных остается в админском контуре и документации.
