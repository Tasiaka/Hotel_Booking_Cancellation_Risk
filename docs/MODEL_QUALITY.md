# Улучшение качества модели MVP

В финальной версии ДЗ №8 модель была усилена без использования leakage-признаков.

## Что изменено

1. Добавлен train-only target/frequency encoding для признаков:
   - `agent`;
   - `company_raw`;
   - `country`;
   - `market_segment`;
   - `distribution_channel`;
   - `deposit_type`.

2. Encoding считается только на train split. Validation и test не участвуют в расчете статистик категорий. Неизвестные категории получают глобальный train prior.

3. Добавлен validation-based model selection: сервис обучает несколько LightGBM-конфигураций и выбирает лучшую по validation ROC-AUC / PR-AUC / Lift@20.

4. Для денежных расчетов сохранена Platt-калибровка вероятности на validation split. Поэтому `expected_loss` считается через `cancellation_probability`, а не через сырой `risk_score`.

## Итоговые метрики

```text
ROC-AUC       = 0.8595
PR-AUC        = 0.6791
Brier         = 0.1409
Precision@20  = 0.6892
Recall@20     = 0.4874
Lift@20       = 2.4375
```

Top-20% списка риска содержит отмены примерно в 2.44 раза чаще, чем случайная выборка.

## Почему это корректно

В проекте сохранен ключевой принцип: прогноз строится после создания бронирования, но до отмены, заезда и финального статуса. Поэтому post-factum поля не используются. Дополнительные категориальные статистики считаются только на train, что не дает модели видеть validation/test target.


## Финансовая приоритизация Non Refund

В публичном датасете `Non Refund` часто получает высокую вероятность отмены. Это не означает, что такие брони должны попадать первыми в рабочий список менеджера: отель сохраняет оплату по невозвратному условию.

Поэтому MVP разделяет две сущности:

```text
cancellation_probability — вероятность отмены
expected_loss = cancellation_probability × booking_value × deposit_loss_factor
```

Для `Non Refund` используется `deposit_loss_factor = 0.00`; для `No Deposit` и `Refundable` коэффициент равен 1.00, то есть вероятность модели не дисконтируется вручную. Такие брони не попадают в основной рабочий список менеджера “К проверке”, потому что высокая вероятность отмены сама по себе не означает высокий финансовый ущерб для отеля. В админском контуре и документации они остаются доступными для проверки, но рабочий экран менеджера показывает только бизнес-действия с денежным приоритетом.
